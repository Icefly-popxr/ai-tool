# 升级标准详解

本文档详细说明知识卡片的等级定义和升级标准。

## 等级概览

| 等级 | 名称 | 核心含义 | 最低要求 |
|------|------|----------|----------|
| **L1** | 初铸 | 通过基础检验 | 通过四铁律，≥1绿灯+1红灯 |
| **L2** | 试炼 | 真实场景验证 | L1 + 在2个真实场景中使用过 |
| **L3** | 淬火 | 跨维度验证 | L2 + 场景跨越≥2个维度差异 |

## L1 - 初铸

### 定义
通过四铁律检验的初始版本，具备基本的可用性。

### 要求
1. **通过四铁律**：
   - ✅ 原子性：一卡一识
   - ✅ 可行动：有明确的行动指南
   - ✅ 可证伪：红灯不为空
   - ✅ 诚实性：绿灯来自真实经历

2. **结构完整**：
   - ✅ 卡片名称、ID、等级、type
   - ✅ 核心洞察（一句话）
   - ✅ 行动指南（可执行）
   - ✅ 至少1个绿灯场景
   - ✅ 至少1个红灯场景

### 典型场景
- 用户第一次分享某个经验
- 经过提炼但尚未在实际中验证
- 仅有理论推断，没有实际使用经历

### 不升级到L2的原因
- 只有1个使用经历
- 使用经历不够详细
- 没有足够的对比数据

## L2 - 试炼

### 定义
已在2个真实场景中验证有效的卡片。

### 要求
1. **L1的所有要求**
2. **至少2个真实使用经历**：
   - 具体的场景描述
   - 实际执行的过程
   - 实际看到的效果
   - 可以对比的差异

### 真实经历的标准
```
✅ 真实经历包括：
· 时间（什么时候）
· 地点/对象（对谁）
· 背景（什么情况下）
· 执行了什么行动
· 得到了什么结果

❌ 虚假/推测经历：
· "可能用在..."
· "应该能..."
· "大概会..."
· "如果当初..."
```

### 升级验证流程

**用户申请升级时，AI必须追问**：
1. "请描述第一个使用场景"
2. "效果如何？和之前相比有什么不同？"
3. "第二个使用场景是什么时候？"
4. "和第一个场景有什么不同？"

**AI判断标准**：
- 两个场景必须有实质差异（不能是同一个场景重复说）
- 场景差异可以是：对象、环境、时机、问题类型等
- 用户必须描述具体效果，不能只是"用了感觉不错"

### L2的典型特征
- 用户能举出具体例子
- 绿灯和红灯的范围更清晰
- 行动指南可能经过微调
- 可能有小的更新和修正

## L3 - 淬火

### 定义
跨越多个维度差异验证过的深度卡片。

### 要求
1. **L2的所有要求**
2. **场景跨越≥2个维度差异**

### 维度差异类型

| 维度 | 示例 |
|------|------|
| **对象差异** | 对不同的人（领导/同事/客户/下属） |
| **环境差异** | 在不同的环境（办公室/远程/外出/会议） |
| **利害差异** | 涉及不同的利益关系（小任务/重要汇报/晋升/谈判） |
| **频率差异** | 偶尔用/经常用/日常用 |
| **复杂度差异** | 简单问题/复杂问题/模糊问题 |
| **紧急程度差异** | 紧急/不紧急 |
| **领域差异** | 工作/生活/学习/社交 |

### L3的判定示例

**✅ 有效升级到L3**：
```
场景1：向直接领导汇报季度工作（利害：个人绩效评估）
场景2：向跨部门总监汇报项目进度（利害：跨团队资源争取）
场景3：在客户演示中介绍方案（对象：外部客户）

→ 跨越了：对象差异 + 利害差异 → 可以升L3
```

**❌ 不能升级到L3**：
```
场景1：向领导A汇报
场景2：向领导B汇报
场景3：向领导C汇报

→ 三个场景都是"向领导汇报"，对象相同，利害相似
→ 没有实质差异 → 不能升L3
```

### L3的典型特征
- 绿灯和红灯的范围非常清晰
- 行动指南经过多次迭代优化
- 用户对适用边界有深刻理解
- 可能有关联卡片的网络结构

## 降级机制

### 可以降级的情况
- 红灯场景在实际使用中被证实
- 绿灯场景被证伪
- 发现新的不适用情况

### 降级流程
1. AI发现卡片有问题时，指出具体问题
2. 用户确认问题属实
3. 更新卡片内容
4. 如果问题严重，降级处理

### 降级原则
- **不随意降级**：没有证据不降级
- **实事求是**：有问题必须指出
- **保持透明**：说明降级原因

## 升级申请流程

### 用户端
1. 用户发送：`/升级 KC-XXX-YY`
2. AI要求用户描述使用经历
3. 用户提供具体场景
4. AI评估是否满足升级条件
5. AI输出结果：升级成功/升级失败

### AI端
1. 收到升级申请
2. 检查卡片当前等级
3. 追问使用经历（至少2个场景）
4. 评估场景差异度
5. 给出升级结论和理由

### 评估标准检查清单

**L1→L2 检查**：
- [ ] 是否有2个以上的真实使用场景？
- [ ] 每个场景是否有具体描述？
- [ ] 场景之间是否有实质差异？
- [ ] 用户描述了实际效果吗？

**L2→L3 检查**：
- [ ] 场景是否跨越≥2个维度差异？
- [ ] 是否涵盖不同类型对象？
- [ ] 是否涵盖不同利害程度？
- [ ] 是否涵盖不同使用频率？

## 常见问题

### Q: 只有一个场景能用，但用了很久，能升L2吗？
**A**: 不能。L2要求至少2个真实场景。如果只有一个场景，可以积累更多经验后再申请。

### Q: 场景相似但用了不同工具，算差异吗？
**A**: 工具差异可以作为参考，但不能作为主要差异。关键是看场景本身是否不同。

### Q: 升级失败后多久可以再申请？
**A**: 没有时间限制。但建议积累更多经验后再申请，不要频繁申请。

### Q: 可以跳级吗？（L1直接升L3）
**A**: 不可以。必须逐级升级。

### Q: L3降级后还能再升回去吗？
**A**: 可以，但需要重新通过验证流程。
